Introduction: The web application takes in geographical information and displays it on screen. It does so by allowing users to post their credentials with a post method, and then it returns a list of all of the previous entries.

Methodology: To test the application, the only external tool I used was curl to input data. I was able to expose the first vulnerability through manual testing, and then delved into the source code to identify two other locations where this vulnerability could be exploited.

Abstract of Findings: The three findings were actually one finding that was exploitable in three sections of the data. The latitude, longitude, and message data fields were all vulnerable to an attack called cross-site scripting. In this attack, it is possible to post malicious HTML or even Javascript as input into the data fields, thereby changing the website's appearance or even redirecting users to entirely different pages.

Issues Found: The issues were all of the cross-site scripting variety found in three separate locations.

  • Issue - Cross-Site Scripting
  • Location - The sendLocation page, specifically in the fields where 1. latitude, 2. longitude, and 3. message are entered.
  • Severity of issue - According to OWASP, cross-site scripting "allows attackers to execute scripts in the victim's browser which can hijack user sessions, deface web sites, or redirect the user to malicious sites." The attack is quite severe because of these possibilities.
  • Description of Issue - I found the issue initially through knowledge of common cross-site scripting attacks, and saw if I could inject HTML through the message data field using curl. I then looked at the source code and found that latitude and longitude had similar issues. The main problem is that the code accepts all types of string values and prints to the screen using HTML, so that the user inputs are modifying the page directly.
